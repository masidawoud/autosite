Per-client Setup Flow

What happens when a new dental practice is onboarded — initially semi-manual, scriptable over time

Onboarding steps

1
Run existing pipeline
node build-sites.js generates site.json via Groq, builds theme. Content files are ready in builds/{id}/
2
Create private GitHub repo
Named e.g. client-tandarts-amsterdam. Push initial content files: site.json, theme.json, sections.json, menu.json, pages/. Add GitHub Action workflow file.
3
Connect to Tina Cloud
Create project in Tina Cloud dashboard, link to client repo. Create client login (email + password). Client never sees GitHub.
4
Add repo secrets
CLOUDFLARE_API_TOKEN, CLOUDFLARE_ACCOUNT_ID, TINA_CLIENT_ID, TINA_TOKEN — all set once per client repo via GitHub API or dashboard.
5
Send client their login
URL: tandartspraktijk.pages.dev/admin. Email + temporary password. Done.

GitHub Action (per client repo)

.github/workflows/deploy.yml
trigger: push to main
    OR repository_dispatch
      (for template updates)

steps:
1. checkout client repo
2. clone dental-template
3. copy content files into template
4. npm install (cached)
5. npm run build
6. wrangler pages deploy
Template update trigger
When operator pushes to dental-template, a script calls GitHub API to dispatch repository_dispatch on all client repos → all GitHub Actions fire → all sites rebuild with updated components. Client content untouched.
Automation potential
Steps 2–4 are fully scriptable via GitHub API (create repo, push files, set secrets, create Tina Cloud project). Initially manual is fine — automate when you're onboarding more than a few clients/week.
