TinaCMS: Cloud vs Self-Hosted

Two ways to run TinaCMS — different trade-offs on cost, complexity, and control

A

Tina Cloud (hosted)

Tina manages the backend — authentication, git operations, content indexing. You just embed TinaCMS in the Astro site and point it at Tina Cloud. No server to run.


How it works
Flow per client
Client logs in via Tina Cloud (email/password) →
Edits content in TinaCMS editor embedded in their site →
Tina Cloud commits changes to client's GitHub repo →
GitHub Action triggers → Astro build → Cloudflare Pages

Pros

  • No server to manage
  • Auth handled out of the box
  • Fastest to get running
  • Scales without effort

Cons

  • $29/month (Starter: 2 users, unlimited sites)
  • Vendor dependency for auth + git ops
  • Less control over user management

Cost: $29/month flat for all clients (2 users per site). Scales well — same price for 5 or 50 clients.

Added complexity vs current setup: Medium — schema definition, template changes, GitHub Action per client. No new infrastructure.

B

Self-Hosted TinaCMS

You run the TinaCMS backend on your own server (a small Node.js process). Handles auth and git operations yourself. TinaCMS open source packages do the heavy lifting — you just configure and deploy.


How it works
Flow per client
Client logs in via your auth (NextAuth / custom) →
Edits content in TinaCMS editor →
Your TinaCMS server commits to client's GitHub repo →
GitHub Action triggers → Astro build → Cloudflare Pages

Pros

  • No platform cost — ~€5–10/month VPS
  • Full control over auth and user management
  • One server handles all clients
  • Foundation for future admin panel

Cons

  • You manage the server (uptime, updates)
  • Auth setup is non-trivial
  • More moving parts to debug
  • TinaCMS self-hosted docs are thinner

Cost: €5–10/month VPS flat. No per-client cost.

Added complexity vs current setup: High — server setup, auth system, per-client configuration, ongoing ops.

What TinaCMS adds to the project regardless of hosting
