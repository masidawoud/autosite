# AutoSite CMS — Sveltia CMS + Gitea Design

**Date:** 2026-03-21
**Type:** Architecture design spec
**Status:** Approved
**Replaces:** Directus (parked), TinaCMS self-hosted (under evaluation, superseded by this spec)

---

## Overview

This spec describes the CMS architecture for AutoSite using Sveltia CMS as the editor frontend and Gitea as the Git backend. Content changes trigger a Gitea Actions pipeline that builds and deploys each client's Astro site to Cloudflare Pages via Wrangler CLI.

The design satisfies the core product requirements:
- Clients log in with email and password only — no GitHub account, no developer knowledge required
- Content is stored in Git — version-controlled and recoverable from source
- Multi-tenancy is structurally isolated — no shared CMS server, no cross-client state
- Template updates propagate to all live sites in a single operator command

---

## Architecture

```
┌─────────────────────────────────────────────────────────┐
│  Operator-owned infrastructure                           │
│                                                         │
│  Gitea instance (self-hosted VPS)                       │
│  ├── dental-template  (operator repo — Astro template)  │
│  ├── client-001       (content: site/theme/sections)    │
│  ├── client-002                                         │
│  └── ...                                                │
│                                                         │
│  act runner (same VPS or dedicated)                     │
│  └── executes .gitea/workflows/deploy.yml per client    │
└─────────────────────────────────────────────────────────┘

┌─────────────────────────────────────────────────────────┐
│  Per client — Cloudflare Pages project                  │
│  ├── /          live Astro site                         │
│  └── /admin     Sveltia CMS (static HTML + config.yml)  │
└─────────────────────────────────────────────────────────┘
```

**Content save flow:**
1. Client visits `client-site.com/admin` — Sveltia CMS loads (static JS)
2. Sveltia shows Gitea OAuth popup — client enters email + password
3. Client edits content — Sveltia commits changes to client's Gitea repo
4. Gitea Actions triggers `deploy.yml` on push
5. Workflow clones `dental-template`, injects client content files, runs `astro build`, deploys via `wrangler pages deploy`
6. Cloudflare Pages serves the updated site

---

## Separation of Concerns

| Repository | Owner | Contains |
|---|---|---|
| `dental-template` | Operator | All Astro components, layout, `/public/admin/index.html` (Sveltia entry point) |
| `client-{id}` | Client (via Sveltia) | `site.json`, `theme.json`, `sections.json`, `admin/config.yml`, `.gitea/workflows/deploy.yml` |

Client repos contain no code — only content JSON files, their CMS config, and their workflow file. The template is never forked per client; it is cloned fresh on every build.

## Authentication

**Client login flow:**
1. Client visits `/admin` on their site
2. Sveltia initiates Gitea OAuth PKCE flow — popup or redirect to the Gitea instance
3. Client enters email + password (Gitea local account, provisioned by operator)
4. Gitea redirects back to Sveltia with an access token — client is in the editor

**Account management:**
- Operator provisions all client accounts via Gitea API or CLI (`gitea admin user create`)
- A single Gitea OAuth application is registered on the operator's Gitea instance; all client CMS deployments reference the same `app_id`
- Per-repo collaborator access: each client has Write access to their own repo only — no visibility into other client repos

**Open question — password reset UX:**
Gitea supports self-service password reset via email natively. However, the flow when the operator provisions an account with a system-set password (rather than the client setting their own on first login) needs to be tested before the provisioning sequence is finalised. Whether `DISABLE_REGISTRATION` is enabled from the start depends on this test.

## Per-Client Provisioning

All steps are scriptable via Gitea API and Cloudflare API. Executed by `build-sites.js` (or a dedicated onboarding script) during client conversion.

1. Create Gitea user account (email, username, initial password) via `POST /api/v1/admin/users`
2. Create Gitea repository `client-{id}` via `POST /api/v1/user/repos`
3. Add Gitea repo secrets (`CLOUDFLARE_API_TOKEN`, `CLOUDFLARE_ACCOUNT_ID`, `CF_PROJECT_NAME`) via `PUT /api/v1/repos/{owner}/{repo}/actions/secrets/{secretname}`
4. Add client user as collaborator with Write access via `PUT /api/v1/repos/{owner}/{repo}/collaborators/{username}`
5. Populate repo with initial content files: `site.json`, `theme.json`, `sections.json` (generated by existing pipeline), `admin/config.yml` (rendered from template with client's repo details), `.gitea/workflows/deploy.yml` (rendered from template with client's CF Pages project name)
6. Create Cloudflare Pages project (existing pipeline behaviour)
7. Push initial content to trigger first build — or operator triggers manually

## Build Workflow

Each client repo contains `.gitea/workflows/deploy.yml`:

```yaml
on: [push]

jobs:
  deploy:
    runs-on: ubuntu-latest
    steps:
      - uses: https://github.com/actions/checkout@v4

      - name: Clone dental-template
        run: git clone https://gitea.yourdomain.com/operator/dental-template.git template

      - name: Inject client content
        run: |
          cp site.json template/src/data/
          cp theme.json template/src/data/
          cp sections.json template/src/data/
          cp admin/config.yml template/public/admin/config.yml

      - name: Build
        working-directory: template
        run: npm ci && npm run build

      - name: Deploy
        working-directory: template
        run: npx wrangler pages deploy dist --project-name=${{ secrets.CF_PROJECT_NAME }}
        env:
          CLOUDFLARE_API_TOKEN: ${{ secrets.CLOUDFLARE_API_TOKEN }}
          CLOUDFLARE_ACCOUNT_ID: ${{ secrets.CLOUDFLARE_ACCOUNT_ID }}
```

**Notes:**
- `dental-template` must be publicly accessible on the Gitea instance (no secrets in template source), or the workflow must use a read-only deploy key to clone it
- `npx wrangler` pulls Wrangler at runtime — pin the version (`npx wrangler@x.y.z`) once validated to avoid silent breakage from upstream changes
- The act runner must have Node.js and Docker available; `ubuntu-latest` runner image is the standard act runner base
- `CF_PROJECT_NAME` is provisioned as a Gitea repo secret during onboarding; using a secret rather than `gitea.repository` avoids the `owner/repo` format that Wrangler rejects as a project name
- `admin/config.yml` is copied from the client repo into `template/public/admin/` before the build so it is included in the Astro output alongside Sveltia's `index.html`; both files must be co-located at `/admin/` in the deployed output

## Sveltia CMS Configuration

Each client's `admin/config.yml` is generated during provisioning:

```yaml
backend:
  name: gitea
  repo: operator/client-001
  base_url: https://gitea.yourdomain.com
  api_root: https://gitea.yourdomain.com/api/v1
  app_id: <operator-oauth-app-client-id>

media_folder: public/images
public_folder: /images

collections:
  - name: global
    label: Global
    files:
      - name: global
        label: Header & Footer
        file: site.json
        fields:
          - name: global
            label: Global
            widget: object
            fields:
              - { name: logo_url, label: Logo URL, widget: string }
              - { name: nav_links, label: Navigation Links, widget: list, fields: [...] }
              - { name: footer_contact, label: Footer Contact, widget: object, fields: [...] }
              - { name: footer_social, label: Social Links, widget: list, fields: [...] }
              - { name: footer_legal, label: Legal Copy, widget: string }

  - name: settings
    label: Site Settings
    files:
      - name: site
        label: Content
        file: site.json
        fields:
          - { name: hero, label: Hero, widget: object, fields: [...] }
          - { name: services, label: Services, widget: list, fields: [...] }
          - { name: team, label: Team, widget: list, fields: [...] }
          - { name: reviews, label: Reviews, widget: list, fields: [...] }
          - { name: contact, label: Contact, widget: object, fields: [...] }
      - name: theme
        label: Theme
        file: theme.json
        fields:
          - { name: colors, label: Colours, widget: object, fields: [...] }
          - { name: fonts, label: Fonts, widget: object, fields: [...] }

  - name: sections
    label: Page Sections
    files:
      - name: sections
        label: Section Order
        file: sections.json
        fields:
          - name: sections
            label: Sections
            widget: list
            allow_reorder: true
            fields:
              - { name: id, label: Section, widget: select, options: [hero, services, team, reviews, contact, faq] }
              - { name: enabled, label: Enabled, widget: boolean }
```

**Field definitions:** Fields shown as `[...]` in the config above are structural placeholders. Full field schemas (labels, widgets, validation) for `site.json` and `theme.json` will be defined as part of the implementation task for the Sveltia config, derived directly from the existing `site.json` and `theme.json` schemas in `dental-template/src/data/`. Clients cannot access raw HTML or CSS — Sveltia operates exclusively on JSON files, so this constraint is structurally enforced.

**Section reordering:** Sveltia supports reordering via arrow handles today. Drag-and-drop is on the Sveltia roadmap but not yet implemented. Arrow-handle reordering is usable for non-technical clients.

**Sveltia version requirement:** Gitea backend requires Gitea 1.24+ or Forgejo 12.0+.

## Template Propagation

When the operator updates `dental-template` and wants all client sites to rebuild:

```bash
node trigger-rebuilds.js
```

The script reads `prospects.csv` (status=completed) and calls the Gitea API per client to trigger a workflow dispatch:

```
POST /api/v1/repos/operator/client-{id}/actions/workflows/deploy.yml/dispatches
```

All client sites rebuild against the new template. Client content files are untouched — they live in the client repo, not in the template.

## New Page Creation

New page creation (clients adding pages beyond the home page) is a **Phase 2 feature**. It requires:

- A `src/pages/[slug].astro` dynamic route in `dental-template` that reads from a pages collection
- A `pages/` entry collection added to the Sveltia config
- The Sveltia List field supports creating new entries natively

This is a template change scoped to Phase 2. Phase 1 is single-page (index only), which matches the current template.

---

## Forms Configuration

Form configuration (contact form recipient email, fields shown, confirmation message) is a **Phase 2 feature**, deferred for the same reason as the forms handling spec notes: static Astro output on Cloudflare Pages has no native form backend. A form solution (e.g. Cloudflare Workers endpoint, third-party provider) must be chosen and validated before the CMS form configuration feature can be designed. Once resolved, a `forms.json` file and corresponding Sveltia collection will be added to the per-client content model.

## Infrastructure Requirements

| Component | Hosting | Notes |
|---|---|---|
| Gitea instance | Self-hosted VPS (post-trial) | Gitea Cloud Enterprise available for 30-day trial ($9.50+/user/month after) |
| act runner | Same VPS as Gitea or dedicated | Requires Node.js + Docker; connects to Gitea instance |
| dental-template repo | Gitea instance | Operator-controlled, publicly readable |
| Client repos | Gitea instance | One per client, Write access to client only |
| Client sites | Cloudflare Pages | One project per client, free tier |
| Sveltia CMS | Cloudflare Pages (at /admin) | Static files shipped as part of dental-template |
| SMTP server | Operator-configured | Required for Gitea to send password reset emails; must be configured on the Gitea instance before any client accounts are provisioned |

**Self-hosting cost:** VPS (e.g., Hetzner CX22, ~€4/month) covers Gitea + act runner for dozens of clients. Per-client cost at scale is effectively zero for Git hosting.

## Open Questions

| Question | Impact | Resolution path |
|---|---|---|
| Gitea password reset UX when operator provisions account | Determines initial account registration policy; SMTP must be configured first | Test: provision via API with SMTP enabled, verify reset email and first-login flow |
| Gitea Actions + Wrangler end-to-end | Confirms CI/CD pipeline before committing | Spike: create test client repo on Gitea trial, push change, verify CF Pages deployment |
| Pin Wrangler version in workflow | Prevents silent breakage from upstream Wrangler changes | Determine once pipeline is validated |
| Single OAuth app across all client repos | Confirms one OAuth app registration is sufficient for all clients | Test: verify Gitea allows one OAuth app used across multiple repos/users |
| Self-hosting Gitea post-trial | VPS selection, act runner setup, SMTP configuration | Architecture decision before go-live; Gitea Cloud Enterprise is $9.50+/user/month after trial |
| Act runner capacity under concurrent template propagation | `trigger-rebuilds.js` dispatches all client workflows simultaneously; on a shared VPS this may cause memory exhaustion or queue delays | Monitor during first full propagation test; consider a dedicated act runner VPS if client count exceeds ~20 |
| Sveltia Gitea backend maturity | Gitea backend is a recent addition to Sveltia; edge cases vs. GitHub backend unknown | Validate core flows (auth, commit, media upload) during the Gitea Actions spike |
| `trigger-rebuilds.js` dependency on `prospects.csv` | Template propagation reads `prospects.csv` for client list — this file must remain the source of truth or be replaced by a Gitea-API-driven client enumeration | Acceptable for Phase 1; revisit if a client database replaces `prospects.csv` |
| `global` and `settings.site` collections both write to `site.json` | Two separate Sveltia file collection entries pointing at the same file may overwrite each other's fields on save (last write wins) if Sveltia does not deep-merge on partial edits | Validate during Gitea Actions spike; clean fix is to collapse into one file collection entry with all fields, or confirm Sveltia merges at field level |
| `allow_reorder: true` on list widget | This key is not in the documented Sveltia list widget API — may be silently ignored or cause a parse warning | Verify against Sveltia schema during implementation; if unsupported, remove it (reordering via arrow handles is the default and works without it) |
